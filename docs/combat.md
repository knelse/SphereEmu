# Боевая система v1 (кулаки + смерть моба)

Справка к PR. Код: `DamageTargetHandler.cs`, `DamageFormula.cs`, `CombatBalance.cs`,
`CommonPackets.cs` (билдеры `FistAttackTargetEcho`, `EntityKilled`), `Monster/Combat.cs`, `Monster/Death.cs`,
`WorldObject/DeathBroadcast.cs`. Конфиг: `Config/Balance/combat.json`.

## Формула урона

Формула предоставлена `akvarel.` ([Discord](https://discord.com/channels/1234553522117283860/1234553522561749056/1521083289396445285)); реализация — `DamageFormula.RollSchoolDamage`:

```
Aavg = (Amin + Amax) / 2
K    = H / Aavg                          // H = урон из статов (PAtk и т.п.)
Roll = uniform(Amin*K, Amax*K)           // E[Roll] = H, оружие задаёт только разброс
Damage = Roll - Defense*(6/7)            если Roll >= Defense
       = Roll^2 / (Defense*7)            если Roll <  Defense
```

Полная формула суммирует школы: `TotalDamage = PhysDamage + MagicDamage`; v1 использует только
физическую часть (маг. школа зарезервирована).

Ветки непрерывны в точке Roll == Defense (обе дают Defense/7); это держится, пока
`defenseSubtractFactor == 1 - 1/defenseQuadraticDivisor` — при тюнинге менять оба ключа вместе.
Guards: Defense <= 0 → урон = Roll; Aavg <= 0 → плоский бросок H без расхода RNG;
Roll <= 0 → 0 (иначе квадратичная ветка возвела бы отрицательный бросок в положительный урон).
Формула чистая (без Godot/I-O/пакетов), RNG инжектится — тестируется с сидом.

Ключи `combat.json` (recovered = восстановлено из оригинала, invented = придумано и тюнится):

| Ключ | Смысл | Происхождение |
|---|---|---|
| `defenseSubtractFactor` | множитель защиты в вычитающей ветке; ровно 6/7 | recovered |
| `defenseQuadraticDivisor` | делитель квадратичной ветки; 7 | recovered |
| `damageClampMax` | 30000 — предел кодирования wire-поля, НЕ геймплейный тюнинг | recovered |
| `fistAminAmax` | [Amin, Amax] кулаков — у кулаков нет строки оружия | invented |
| `fistStatSheetDamage` | плоская добавка к \|PAtk\| (якорь: кулаки в оригинале били ~2; PAtk без MainHand → у голого перса 0) | invented |
| `minMeleeHit` | пол урона при попадании (иначе кулаки против PDef >= 1 навсегда 0) | invented |
| `roundingMode` | "floor" / "round"; floor соответствует клиентскому int(...) | invented |
| `critChance` / `critMult` / `missChance` | крит/промах на уровне хендлера; по умолчанию выключены — свидетельств их существования в оригинале нет | invented |
| `meleeRangeMeters` | серверная страховка дальности (10.0, клиент сам держит ~1.5); <= 0 отключает | invented |

Порядок в `RollMeleeHit`: бросок промаха → формула (H = |PAtk| + fistStatSheetDamage) →
крит с повторным клампом к `damageClampMax` → пол `minMeleeHit`. Порядок RNG фиксирован
ради детерминизма тестов. Промах даёт 0, но ответный фрейм всё равно уходит.

## Разбор атакующего фрейма (first byte 0x19/0x20)

Не-buy фреймы этого маршрута падают в `DamageTargetHandler`; различаются дискриминатором
в байтах 13–15 (`ParseAttackFrame`):

| Байты | Что это | Действие сервера |
|---|---|---|
| `08 40 A3` (13–15) | левый клик — атака кулаками (`FistAttack`) | формула + echo |
| `08 40 83` (13–15) | правый клик — interact/enter (двери, объекты) | НИКАКОГО ответа |
| `54 43 C1` (13–15) | self-target через Alt (`SelfTargetedAction`): в битах 172–187 id самого атакующего | log-and-drop |
| `7E 14 CE` (22–24) | атака оружием (`WeaponAttack`); смещение цели не разобрано | echo 0 урона |

Id цели — u16 LSB-first после 172 пропущенных бит (байты 21–23); длины маршрута
(0x19 = 25, 0x20 = 32 байта) всегда его вмещают. Ответ атакующей формы на не-атаку —
кривой фрейм, который может поломать клиентскую обработку интеракций, поэтому
`NotAnAttack` — строгий no-op. Левый клик по трупу клиент маршалит в `08 40 83`;
его use-lock снимает ack в начале `Handle`. `SelfTargetedAction` нельзя парсить как атаку —
echo прилетел бы в самого атакующего.

Отдельный случай: атака без цели (`0xFFFF` в поле цели — клик по месту, где моб уже
исчез). На неё НЕЛЬЗЯ отвечать: entity-адресованный фрейм по несуществующему id роняет
скриптовую VM клиента (BoundCheckArray в `_player`; воспроизведено вживую). Хендлер её
молча поглощает (use-ack при этом уходит как обычно).

## Ответ на удар

`CommonPackets.FistAttackTargetEcho` — S→C `0x15`, 21 байт, ровно один на каждый
запрос атаки: swing-цикл клиента ждёт именно его. Поле урона кодируется как
`30000 - damage` (клиент применяет `raw - 30000` к HP цели), отсюда и жёсткий
диапазон [0, 30000]. В echo идёт ПРИМЕНЁННЫЙ (клампнутый сервером) урон, чтобы
дельта HP на клиенте совпадала с серверной. `object_type` всегда 210 (Monster) —
так вёл себя живой сервер; сверено побайтово с живой сессией.

Политика echo: живой Monster в радиусе → реальный урон; неизвестная/мёртвая/не-Monster
цель или вне радиуса → echo 0 урона (свинг рисуется, состояние не меняется).

Первая инструкция `Handle` — отправка `ClearUseToutAck` (фикс attack-wedge, апстрим PR #11):
до любого парсинга и ранних return'ов, чтобы ack не терялся ни на одном drop-пути.

## Смерть моба

`TakeDamage` (`Monster/Combat.cs`) чистый по сети — пакетов не шлёт; `Applied`
клампится к [0, CurrentHp], `BecameDead` истинно ровно один раз (переход HP > 0 → 0),
удар по трупу — полный no-op. Пайплайн `PerformDeath` (идемпотентен, только main thread):

1. `SetPhysicsProcess(false)` — стоп нав/AI/рассылки позиции;
2. `BroadcastDeathSignalToVisibleClients` — `entity_killed` каждому наблюдателю;
3. отложенно на тик (`CallDeferred`): деспаун + `RemoveFromWorldRegistry`
   (`ActiveWorldObjects` + `ActiveNodes`, иначе висят на freed node) + `QueueFree`
   (для мобов под `MonsterSpawner` именно он взводит респаун).

ВАЖНО про порядок: ответный фрейм с уроном хендлер кладёт в очередь ПОСЛЕ `TakeDamage`,
поэтому деспаун отложен на тик — клиент должен получить ответный фрейм с уроном (fist_attack_target) раньше деспауна.
Дельта урона по уже удалённой сущности роняет скриптовую VM клиента
(BoundCheckArray в `_player`; воспроизведено вживую).

Фрейм `entity_killed`: шаблон `.spdp` с зашитыми `action_type` INTERACT (0x0A) и
`interaction_type` DEATH (0x040D); переопределяются только `entity_id` и `killed_by_id`
(client-local, пока identity-mapped global == local — поэтому фрейм строится per client).
84 контентных бита, 4-битный хвост клиентский сканер принимает. `killed_by_id = 0` —
смерть без атрибуции. Клиент по этому фрейму запускает программу смерти: стоп AI,
анимация, fade. Немедленный деспаун сразу за сигналом безопасен — клиент доигрывает
анимацию полностью (проверено на живом клиенте). Без этого фрейма моб на 0 HP
стоит столбом с пустым HP-баром; ту же картину даёт клиент, вошедший в радиус
после смертельного удара — поэтому `ShowForClient` гейтится по `_deathStarted`
(деспаун никогда не заспавненной сущности для клиента безвреден).

## Точки расширения

- `Damaged` — после каждого применения урона (включая промахи-нули); для aggro/threat AI.
- `MonsterKilled` — на переходе в смерть, до `PerformDeath`; для XP/лута/наград.
- `DamageSchool.Magical` зарезервирован (положительный сырой MAtk = хил, в melee не попадает).

## Ограничения v1

- Только кулаки: у CUse-атаки оружием не разобрано смещение цели (в битах 172–187 мусор) — пока echo 0 урона.
- Порошки и атакующие мантры не классифицированы: их фреймы никогда не встречались в captures (нечем атаковать без инвентаря). Порошки экипируются в mainhand как оружие,
  так что скорее всего идут тем же CUse-путём; атакующие мантры — неизвестная форма.
  Любой нераспознанный фрейм безопасно поглощается без ответа.
- Нет XP/лута/кармы — хуки выше готовы, подписчиков нет.
- Крит/промах выключены по умолчанию (нет свидетельств из оригинала).
- `combat.json` читается из дерева репозитория: подъём вверх от `AppContext.BaseDirectory` /
  `Environment.CurrentDirectory` (Godot собирает в `.godot/mono/temp/bin`), затем
  fallback на `ServerConfig.AppConfig.RepositoryPath` — для экспортнутого билда
  конфиг нужно будет паковать отдельно.
